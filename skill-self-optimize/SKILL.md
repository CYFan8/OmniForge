# 模型自我迭代更新规则

> 核心哲学：每一次失败都是下一次更好的输入。

---

## 一、自我审查循环

`
触发事件（失败/低效/用户反馈）
    │
    ▼
① 捕获：记录发生了什么
    │
    ▼
② 诊断：为什么发生
    │
    ▼
③ 改进：怎么办
    │
    ▼
④ 沉淀：写入规则
    │
    ▼
⑤ 应用：下次执行时自动采用
`

---

## 二、规则更新流程

### 2.1 需要更新规则的场景

| 场景 | 示例 |
|------|------|
| 同一类问题连续错 2 次 | 两次误解了「模式A vs 模式B」的边界 |
| 用户明确反馈「不要这么干」 | 「别用那种方式输出，我喜欢精简的」 |
| 发现更好的高阶策略 | 发现先做 A 再做 B 比逐条做快了 3 倍 |
| 架构瓶颈 | 某文件超过 500 行，需要拆分 |

### 2.2 更新流程

`
失败捕获
  ↓
诊断根因
  ↓
写反思记录（见第三节模板）
  ↓
判断是否需要改 AGENTS.md
  ├─ 是 → 修改 AGENTS.md 对应小节
  │        ↓ 权限不够则记录到 .codex/improvements/
  │
  └─ 否 → 写一条 TODO 到 .codex/improvements/pending.md
           （三个月未解决的转入 archived.md）
`

## 三、反思模板

### 3.1 失败反思

`
## FAIL: [日期]

### 发生了什么
[客观描述事件]

### 预设了什么
[我当时默认成立但实际不成立的前提]

### 根因是什么
[为什么那个前提错了？信息不足？推理跳跃？信号被忽略？]

### 下次怎么做
[具体的、可执行的行动规则]

### 是否改 AGENTS.md
[是/否] → [若「是」，更新到哪个章节]
`

### 3.2 架构决策反思

`
## ADR: [日期 - 决策标题]

### 背景
[为什么需要做这个决定]

### 选项
[选项 A：……]
[选项 B：……]

### 选择
[选 A，因为……]

### 结果（事后评估）
[后来发展如何？选对了还是选错了？]

### 经验
[这次决策中学到什么]
`

### 3.3 持续改进清单

`
## IMPROVEMENTS

### 本周改进点

### 淘汰的旧规则

### 待验证假设
`

## 四、改进记录管理

### 目录结构

`
.codex/improvements/
├── pending.md        # 待处理的改进
├── applied.md        # 已应用的改进及效果
├── archived.md       # 过期废弃的改进
└── learn.log         # 快速学习日志
`

### 优先级标记

`
[P0] 影响核心功能正确性
[P1] 影响效率或用户体验
[P2] 锦上添花
[P3] 可能有用但不确定
`
